# MOY 价值归因 增量升级方案

> 目标：在现有架构基础上，构建"结果证明"，让客户看到可量化价值，提升付费续费意愿

---

## 一、现状判断

### 1.1 已具备结果归因基础的能力

| 数据层 | 结构 | 现有能力 | 价值归因潜力 |
|------------ |---------------------------|------------------------------------------------------ |----------------- |
| 行动结果 | `action_outcomes` | 记录每次行动的结果、阶段变化、关键产出 | ⭐⭐⭐⭐⭐ 核心 |
| 建议采纳 | `suggestion_adoptions` | 追踪 prep_card/content_draft 的使用情况，可关联 outcome | ⭐⭐⭐⭐ ⭐ |
| 业务事件 | `business_events` | 自动化规则生成的事件，包含状态流转 | ⭐⭐⭐⭐ ⭐ |
| 健康度快照 | `customer_health_snapshots` | 每日健康度评分、风险标记、正向信号 | ⭐⭐⭐⭐ ⭐ |
| 自动化执行 | `automation_rule_runs` | 规则执行记录、匹配数、创建动作数 | ⭐⭐⭐⭐⭐ |
| 任务执行日志 | `task_execution_logs` | 任务状态变更记录 | ⭐⭐⭐⭐⭐ |
| 高管简报 | `executive_briefs` | AI 生成的简报内容 | ⭐⭐⭐⭐ 需增强 |

**结论**：数据层基础已经具备，关键问题在**展示层和归因逻辑不完整**

| 服务 | 文件 | 核心功能 | 复用价值 |

| `customer-health-service.ts` | 计算健康度、生成快照 | ⭐⭐⭐⭐⭐ 直接复用 |
| `automation-rule-service.ts` | 规则执行、动作创建 | ⭐⭐⭐⭐ 直接复用 |
| `work-item-service.ts` | 任务创建/状态变更、执行日志 | ⭐⭐⭐⭐ 直接复用 |
| `executive-brief-service.ts` | 生成高管简报 | ⭐⭐⭐ 需增强 |

**结论**：服务层架构成熟，无需大规模重构，主要工作是**增强归因查询和聚合逻辑**

### 1.3 现有页面能力

| 页面 | 路由 | 现有功能 | 结果展示现状 | 增强空间 |
|----------|----------------------|--------------------|--------------|-------------|
| 今日工作 | `/today` | 展示今日任务、提示 | 无价值汇总 | ⭐⭐⭐⭐⭐ 增强 |
| 早报 | `/briefings` | 展示早报内容 | 无归因展示 | ⭐⭐⭐⭐⭐ 增强 |

## 二、问题清单

### 2.1 关键缺失环节

#### 问题 1：价值归因链路不完整

**现象**：
- 系统会 `action_outcomes` 记录行动结果
- 系统会 `suggestion_adoptions` 记录建议采纳
- 但两者之间缺少数**从问题发现到结果展示**的可视化链路

**影响**：
- 客户看不到**系统建议 → 我采纳 → 产生结果**的完整故事
- 无法证明 AI 建议的实际价值

**现有数据支撑**：
- `suggestion_adoptions.linked_outcome_id` 已存在，但未被充分利用
- `action_outcomes.work_item_id` 可追溯到 `work_items.source_ref_type = 'business_event'`

#### 问题 2：效率提升缺少量化展示

**现象**：
- `/today` 页面只展示任务列表，不展示**今日完成了多少**
- 没有"节省时间"的概念
- 没有"AI 帮您做了什么"的汇总

**影响**：
- 客户感知不到效率提升
- 无法回答"这个系统帮我省了多少时间"

**现有数据支撑**：
- `work_items` 有 `ai_generated` 字段
- `task_execution_logs` 有执行时间记录
- `action_outcomes` 有 `used_prep_card` / `used_draft` 字段

#### 问题 3：漏跟进缺少可视化

**现象**：
- `business_events` 有 `no_recent_touchpoint` 事件
- `customer_health_snapshots` 有 `followup_inactive` 风险标记
- 但没有**漏跟进预警 → 处理 → 结果**的展示

**影响**：
- 客户不知道系统帮他们避免了多少漏跟进
- 无法证明系统的预警价值

**现有数据支撑**：
- `business_events.event_type = 'no_recent_touchpoint'`
- `customer_health_snapshots.risk_flags` 包含 `followup_inactive`
- `work_items` 可关联到客户

#### 问题 4：客户推进缺少可视化

**现象**：
- `action_outcomes` 有 `stage_changed` / `old_stage` / `new_stage`
- 但没有**本周推进了多少客户**的汇总展示

**影响**：
- 客户看不到销售推进的成果
- 无法感知"系统帮我推进了客户"

**现有数据支撑**：
- `action_outcomes.stage_changed = true` 的记录
- `customers.current_stage` 的变化

#### 问题 5：管理成本缺少量化

**现象**：
- `automation_rule_runs` 有 `matched_count` / `created_action_count`
- 但没有**自动化规则帮您处理了多少事项**的展示

**影响**：
- 管理者看不到自动化的价值
- 无法证明"系统降低了管理成本"

**现有数据支撑**：
- `automation_rule_runs` 的执行记录
- `business_events.status` 的状态流转

### 2.2 数据流和命名一致性问题

| 问题 | 现状 | 影响 | 优先级 |
|--------------------------------------------------- |--------------------------------|-------------------- | ------ |
| `suggestion_adoptions.linked_outcome_id` 使用率低 | 大部分记录该字段为 null | 归因链路断裂 | P1 |
| `work_items.source_ref_type` 缺少统一枚举 | 用 `alert`、`business_event` 等 | 查询复杂 | P2 |
| `action_outcomes` 与 `business_events` 缺少直接关联 | 只能通过 `work_item_id` 间接关联 | 归因查询复杂 | P1 |
| 健康度趋势缺少历史对比 | 只有快照，无趋势计算 | 无法展示"健康度提升" | P2 |

### 2.3 页面展示问题

| 页面 | 问题 | 影响 |
|-------------------- | -------------------- | ------------------------ |
| `/today` | 缺"今日价值"汇总卡片 | 用户看不到今日成果 |
| `/briefings` | 缺"建议采纳"统计 | 用户看不清 AI 价值 |
| `/executive` | 缺"自动化效果"面板 | 管理者看不到 ROI |
| `/settings/automation` | 缺"规则效果"统计 | 无法评估规则价值 |
| `/customers/[id]` | 缺"价值归因"面板 | 看不到系统对该客户的贡献 |

## 三、分 Sprint 计划

### Sprint 1：价值可视化 (2 周)

**目标**：让客户在关键页面看到**系统帮我做了什么**的量化汇总

#### 3.1.1 今日工作页面增强

**新增内容**：

1. "今日价值"汇总卡片
   - 今日完成任务数
   - AI 辅助完成任务数
   - 使用 prep_card / content_draft 的次数
   - 预估节省时间（基于任务类型估算）

2. "待处理预警"卡片
   - 来自 `business_events` 的 open 事件
   - 按严重程度排序

#### 3.1.2 早报页面增强

**新增内容**：

1. "本周价值"汇总卡片
   - 本周完成任务数
   - 本周客户推进数（stage_changed = true 的数量）
   - 本周建议采纳率
   - 本周处理的风险预警数

2. "建议效果"面板
   - 展示 prep_card / content_draft 的使用统计
   - 展示采纳后产生的 outcome 结果

#### 3.1.3 高管驾驶舱增强

**新增内容**：

1. "自动化效果"面板
   - 本周规则执行次数
   - 本周自动创建的任务数
   - 本周处理的风险事件数
   - 自动化覆盖率（自动创建任务 / 总任务）

2. "团队执行效率"面板
   - 平均任务完成时间
   - 漏跟进率（有 risk_flag 的客户 / 总客户）
   - 跟进及时率

#### 3.1.4 Sprint 1 验收标准

**功能验收**：
- [ ] `/today` 页面展示"今日价值"汇总卡片
- [ ] `/briefings` 页面展示"本周价值"汇总卡片
- [ ] `/executive` 页面展示"自动化效果"面板
- [ ] 所有数据基于现有表，无新增 migration

**回归验收**：
- [ ] 现有功能不受影响
- [ ] 页面加载时间 < 2s

**边界验收**：
- [ ] 无数据时展示空状态
- [ ] 权限边界正确（销售只看自己的数据，管理员看全部）

---

### Sprint 2：结果归因与动作闭环 (2 周)

**目标**：构建**问题识别 → 动作执行 → 处理记录 → 结果归因 → 价值展示**的完整链路

#### 3.2.1 价值归因服务层

**新增内容**：

1. `value-attribution-service.ts`
   - `getCustomerValueAttribution(customerId)` - 获取单个客户的价值归因
   - `getOrgValueAttribution(orgId, period)` - 获取组织级别的价值归因
   - `linkOutcomeToBusinessEvent(outcomeId, eventId)` - 关联 outcome 到业务事件

**归因逻辑**：

```
业务事件 (business_events)
    ↓ 触发
工作任务 (work_items)
    ↓ 执行
行动结果 (action_outcomes)
    ↓ 关联
建议采纳 (suggestion_adoptions)
    ↓ 展示
价值归因面板
```

#### 3.2.2 客户详情页价值归因面板

**新增内容**：

1. "价值归因"面板（侧边栏或 Tab）
   - 系统为该客户发现的问题
   - 触发的动作
   - 执行的结果
   - 客户阶段推进历史

#### 3.2.3 自动化规则效果追踪

**新增内容**：

1. 规则效果统计卡片
   - 每条规则的执行次数
   - 每条规则创建的任务数
   - 每条规则处理的事件数
   - 规则 ROI 估算

2. 规则执行详情
   - 点击规则可查看执行历史
   - 每次执行创建的任务列表
   - 任务完成情况

#### 3.2.4 Sprint 2 验收标准

**功能验收**：
- [ ] `/customers/[id]` 页面展示"价值归因"面板
- [ ] `/settings/automation` 页面展示规则效果统计
- [ ] 归因服务正确计算价值指标
- [ ] Migration 成功执行，历史数据回填完成

**回归验收**：
- [ ] 现有功能不受影响
- [ ] Migration 向后兼容

**边界验收**：
- [ ] 无关联数据时展示空状态
- [ ] 权限边界正确

**目标**：为中国企业用户提供更符合本地习惯的工作入口，并为后续通知集成做准备

## 五、建议优先执行顺序

## 六、总结

### 6.1 核心升级方向

1. **价值可视化**：让客户在关键页面看到**系统帮我做了什么**
2. **结果归因**：构建**问题 → 动作 → 结果**的完整链路
3. **中国市场适配**：工作时间、语言风格、通知渠道

### 6.2 复用现有能力

- **数据层**：100% 复用现有表结构，仅新增 2 个字段和 2 个新表
- **服务层**：90% 复用现有服务，仅新增归因服务
- **页面层**：90% 复用现有页面，仅增加组件和面板

### 6.3 预期效果

| 指标 | 当前 | 目标 |
|------------------------------ | ---- | ---- |
| 用户能看到"今日价值" | 否 | 是 |
| 用户能看到"系统帮我推进了客户" | 否 | 是 |
| 管理者能看到"自动化效果" | 否 | 是 |
| 归因链路完整度 | 30% | 90% |
| 中国市场适配度 | 60% | 85% |
