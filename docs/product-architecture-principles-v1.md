# MOY 产品架构原则 v1（内部）

更新时间：2026-03-21  
适用范围：MOY 后续全部产品与研发需求评审、方案设计、实现验收

---

## 1. MOY 的产品定位

MOY 是一个面向中小企业销售团队的销售推进系统，不是“聊天入口”，而是“业务推进操作系统”。

当前代码已明确体现以下定位：

- 首页对外表述是 SMB 销售团队工作台，强调从沟通输入到执行、风控、管理干预、转化复盘的一体化流转（`app/page.tsx`）。
- README 定义的主链路是“发现问题 -> 触发动作 -> 记录处理 -> 归因结果 -> 展示价值”，并已落地到 capture / today / deals / executive / automation 等模块（`README.md`）。
- AI 服务不是自由问答，而是场景化执行引擎：`followup_analysis`、`customer_health`、`leak_risk_inference`，并把结果写回业务对象（`services/ai-analysis-service.ts`）。
- 公域线索与 demo 申请直接进入组织化销售流程（lead -> demo_request -> work_item），不是独立营销系统（`app/api/public/request-demo/route.ts`、`services/demo-request-service.ts`）。

结论：MOY 的核心交付是“推进结果可见性”，不是“AI 体验炫技”。

---

## 2. MOY 不是在卖什么

### 2.1 不是通用聊天工具

- 不提供开放式通用对话产品定位。
- AI 调用以固定业务场景为单位，输出必须符合 schema 并落到结构化字段。
- 当 AI 不可用时，系统回退到规则引擎，保证业务主链路继续写入和执行。

### 2.2 不是单一行业 CRM

- 当前仓库已具备行业模板入口与模板层测试，说明产品方向是“跨行业通用底座 + 行业模板扩展”，不是只服务单一行业。
- CRM 只是数据载体，MOY 的核心价值在“推进动作、风险识别、管理干预、结果归因”。

## 3. MOY 的三层结构

### 3.1 通用销售底座（必须统一）

定义：所有客户都共享、不可按企业随意改写的核心能力层。

已落地能力：

- 组织与权限边界：Supabase Auth + org membership + seat status 与角色判定（`services/auth-service.ts`、`lib/org-membership-utils.ts`、`types/auth.ts`）。
- 销售主流程：沟通采集、跟进任务、商机推进、结果记录、管理视图。
- AI 业务引擎：场景化分析、prompt version、run audit、fallback。
- 自动化运营层：automation rules、business events、executive cockpit、health snapshots。
- 可验证链路：`tests/golden-path-smoke.test.ts` 把捕获、风控、任务、自动化、管理视角、结果归因串成一条“可运行价值链”。

不可破坏约束：

- 不允许绕过 service 层直接在页面里写数据库。
- 不允许让关键业务链路依赖“AI 一定成功”。
- 不允许为单个客户改写底座状态机（如事件状态流、任务状态流、角色基础语义）。

### 3.2 行业模板 / 打法包（可扩展）

定义：在通用底座不变前提下，对行业差异进行复用化封装。

可承载内容：

- 行业话术模板、推进节奏模板、风险规则阈值模板。
- 行业化的 briefing / playbook / action suggestion 内容层。
- 行业线索字段映射和导入模板。

实现要求：

- 通过配置与模板注册实现，不复制一套新业务流程。
- 模板只影响“参数、文案、阈值、推荐策略”，不影响底座数据模型核心关系。

### 3.3 企业定制层（受控定制）

定义：企业级差异化诉求的可配置层，必须在可维护边界内实现。

允许方式：

- 组织级别配置（feature flag、自动化规则开关、配额与计划、角色分配）。
- 组织级 prompt 版本选择（按 `orgId + scenario + provider` 管理）。
- 公域线索入口字段扩展与映射（在现有 lead/demo/work_item 流程中扩展）。
- 报表视角与指标口径的“新增维度/筛选”，不改核心指标定义。

## 4. 哪些能力必须跨行业通用

以下能力必须作为产品统一能力，不接受行业特化分叉实现：

- 组织隔离与权限安全：org 边界、seat 状态、基础角色控制。
- 核心对象与关系：customer、followup、opportunity、alert、work item、business event。
- 风险与推进主链路：识别风险 -> 触发动作 -> 执行记录 -> 状态更新 -> 管理可见。
- AI 运行治理：schema 校验、run 审计、失败回退、结果来源标记（provider/fallback）。
- 自动化事件总线与状态流：`open -> acknowledged -> resolved/ignored`。
- 质量门禁：lint / test / build 全量通过。

---

## 5. 哪些差异通过模板解决

以下需求优先通过“行业模板/打法包”解决，而不是新建功能分支：

- 行业术语与文案风格差异（如制造业/教育/服务业表述）。
- 不同行业的阶段关注点、风险阈值、推荐动作优先级。
- 不同行业 morning brief / manager brief 的信息组织方式。
- 导入字段映射模板（不同来源表头规范）。
- 行业常见 objection 与 next best action 的知识包。

判定原则：

- 如果差异只改变“内容和参数”，必须走模板。
- 只有当差异要求新增底座能力时，才进入底座演进评审。

---

## 6. 哪些需求允许企业定制

允许定制（且应优先配置化）：

- 自动化规则开关、执行频率、告警阈值、动作模板。
- 组织内角色与座席管理策略（在现有角色体系内配置）。
- 组织级 AI 策略开关（例如是否启用某场景自动分析）。
- 导入规则、字段映射、去重策略参数。
- 公域线索采集字段的增补（保持反垃圾与频控机制）。

允许但需评审后落地：

- 新增企业专属报表维度。
- 新增企业专属模板类型。
- 与企业现有系统做 API 对接（仍需经过 service 层，不允许页面直连）。

---

## 7. 哪些需求明确不接受定制

以下诉求默认拒绝：

- 把 MOY 改造成通用聊天助手或开放式问答工具。
- 为单一企业复制一套独立代码分支和独立业务流程。
- 删除或绕过 fallback，要求 AI 失败时中断核心业务写入。
- 绕过 org 隔离、绕过角色权限做“超级查看”。
- 为单企业修改核心状态机语义（事件状态、任务状态、角色基础定义）。
- 在页面层直接访问数据库，规避服务层。

## 8. 架构演进原则（优先参数化、配置化、模块化）

### 8.1 参数化优先

- 先做阈值、权重、文案、策略参数化，再考虑新增流程。
- 先检查现有 scenario 是否可复用，避免新增平行 AI 场景。

### 8.2 配置化优先

- 一切企业差异优先落在组织配置、模板配置、规则配置。
- 配置必须有默认值、审计记录、回滚路径。

### 8.3 模块化优先

- 复用现有 `services/*`、`lib/*`、`types/*` 分层，不在页面堆业务逻辑。
- 新能力优先扩展现有模块（如 automation、briefing、health），避免另起“第二套系统”。

### 8.4 当前已暴露的架构风险（具体）

- 角色语义被压缩风险：`types/auth.ts` 只有 `sales | manager`，而 membership 实际有 owner/admin/manager/sales/viewer；`services/auth-service.ts` 把 owner/admin/manager 都映射为 manager，容易导致权限表达不精确。
- 商机阶段语义一致性风险：`createDemoRequest` 创建 `demo_status=pending` 时会把 lead 状态推进到 `demo_scheduled`，存在“请求已提交”和“已排期”语义重叠风险。
- 商业化入口单点配置风险：`/api/public/request-demo` 强依赖 `SELF_SALES_ORG_ID`，缺失时直接 503；运维配置错误会让公域获客入口整体不可用。
- 集成回归覆盖不足风险：当前 `tests/run-tests.ts` 与 `golden-path-smoke` 以单测与规则链路为主，真实 API + RLS + migration 的端到端回归覆盖仍偏弱。

## 9. 后续开发决策 Checklist

每个需求评审必须逐项勾选：

- [ ] 该需求是否直接服务“推进结果可见性”，而非单纯功能堆叠？
- [ ] 是否优先复用了现有底座模块，而不是新增平行系统？
- [ ] 是否能通过模板/配置解决，而不是写死企业分支逻辑？
- [ ] 是否保持 service 层边界与 org 权限边界？
- [ ] 是否定义了 AI 失败回退路径，并保证主链路不中断？
- [ ] 是否提供了最小可验证测试（单测或 smoke）？
- [ ] 是否同步更新文档，且文档描述与当前代码一致？
- [ ] 是否明确了不会被接受的定制范围，避免范围蔓延？

## 下一阶段最重要的 5 个建设方向

1. 权限模型精细化与一致性收敛  
目标：打通 `org_memberships` 真实角色与前端用户角色表达，避免 owner/admin/manager 被压缩后带来的策略歧义。

2. 模板中心产品化（行业包生命周期）  
目标：形成“模板定义 -> 版本发布 -> 灰度启用 -> 效果评估”的闭环，而不是散落配置。

3. 自动化规则运营化  
目标：把规则命中率、误报率、动作完成率、结果归因统一进运营看板，支持规则级持续优化。

4. 公域获客链路韧性增强  
目标：降低 `SELF_SALES_ORG_ID` 配置错误导致的入口不可用风险，补充健康检查与告警。

5. 真实端到端回归能力建设  
目标：在现有单测基础上增加关键 API + RLS + migration 的集成测试，确保版本迭代不破坏业务闭环。
